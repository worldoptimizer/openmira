v1.7.0 - 2026/05/28
* New: Dynamic/server-rendered Gutenberg block-level editing. Agents can call `openmira/read-blocks` to inspect a post as a side-effect-free block tree with stable refs, then call `openmira/patch-blocks` to update, insert, or delete dynamic/server-rendered blocks atomically with ETag conflict checks. Static/core block editing returns `block_runtime_required` and is coming in a follow-up via the Block Editor Runtime; use `write-gutenberg-content` when a full-page rewrite of static content is explicitly intended.
* New: Block hunks in `openmira/apply-patch`. The patch grammar now supports `*** Update Block`, `*** Insert Block`, and `*** Delete Block` hunks routed through the same dynamic/server-rendered block patch engine.
* New: Async WP-CLI jobs. `openmira/run-wpcli` accepts `mode: "async"` for allowlisted long-running commands, and `openmira/get-wpcli-job` returns incremental logs and completion status.
* New: Built-in `block-editing` Skill documenting the safe read-first, ref-targeted Gutenberg editing workflow.
* Improvement: `openmira/patch-gutenberg-blocks` remains available for backwards compatibility but is demoted from compact discovery in favor of the ref-based block editing surface.
* Security: `openmira/run-wpcli` is now marked destructive and refuses to run as root unless `OPENMIRA_WPCLI_ALLOW_ROOT` is explicitly defined.
* Security: Hardened filesystem path handling. Directory boundary checks now require a true path-segment boundary (fixing a sibling-prefix bypass), writes and deletes through symlinked final paths are rejected, and not-yet-existing paths are normalized to collapse "../" before the boundary check. No configuration change required.
* Note: Static/core block patching intentionally returns `block_runtime_required` in this release. The browser-backed Block Editor Runtime is deferred rather than risking invalid Gutenberg saved markup.
* Minor fixes.

v1.6.1 - 2026/05/23
* New: Added a built-in `skill-creator` Skill, adapted from Anthropic's Apache-2.0 skill-creator source for Open Mira's flat WordPress Skill format.
* Improvement: Custom Skills now use native WordPress trash semantics. Admin users can move CPT-backed Skills to trash, review trashed Skills, restore them, or permanently delete them.
* Improvement: Skills import now accepts multiple Markdown files in one upload and reports per-file import results.
* Improvement: Built-in Skill frontmatter now includes `enable_prompt: true`, and frontmatter `enable_prompt: false` suppresses MCP Prompt registration when no CPT meta exists.
* Minor fixes.

v1.6.0 - 2026/05/22
* Security: Custom Skills now store user-created prompt content in a private WordPress CPT (`openmira_skill`) instead of writing prompt files under `wp-content/openmira-skills/`. This removes the writable-filesystem prompt injection surface from the Skills admin.
* New: Custom Skills now get native WordPress revisions. Admin users can open the native revision viewer for CPT-backed Skills after revisions exist.
* New: Per-skill prompt enablement. CPT-backed Skills can be disabled from MCP Prompt registration without deleting their content.
* New: Source abstraction for Skills. Built-in filesystem Skills and CPT-backed custom Skills are assembled through filterable sources, allowing plugins to register additional backends.
* Improvement: 1.5.x filesystem Skills migrate automatically into CPT storage on first 1.6.0 boot. Legacy files are left in place with a deprecation notice for one minor version, but are no longer loaded as prompt sources.
* Improvement: Skills import/export keeps the stable `SKILL.md` and ZIP interchange format while reading and writing CPT-backed custom Skills.
* Minor fixes.

v1.5.1 - 2026/05/20
* Fix: Skills admin form no longer triggers a browser console error on Chrome (`Invalid character in character class`). The skill ID pattern attribute uses an unambiguous character class compatible with modern browser regex parsers.
* Fix: Skills admin form no longer triggers "An invalid form control is not focusable" on submit. The Markdown editor now syncs its content to the underlying textarea before form submission, and required-field validation is enforced server-side instead of client-side on hidden controls.
* Improvement: Skills admin uses the WordPress wp-list-table layout to match the Memory admin page. Inline styles removed; visual styling consolidated in the shared admin CSS file.
* Improvement: Customizing a built-in skill goes directly to the edit form (one click) instead of routing through the list (two clicks).
* Improvement: Skill edit form has a Cancel button and groups fields into Identity / Metadata / Content fieldsets.
* Minor fixes.

v1.5.0 - 2026/05/20
* New: Skills are now editable from the Open Mira admin. Create custom Skills, override built-in ones, export to ZIP, and import single SKILL.md files or full ZIP archives. Custom Skills live under wp-content/openmira-skills/ and survive plugin updates.
* New: Memory import (companion to the existing Memory export). Paste or upload a previously exported JSON file to restore or seed memory entries.
* New: Markdown editor (CodeMirror with line wrap) for Memory values and Skill bodies in the admin.
* Improvement: Unified visual styling between the Memory and Skills admin pages — consistent toolbar, action links, and source badges.
* Minor fixes.

v1.4.0 - 2026/05/19
* New: Skills (MCP Prompts) infrastructure. Open Mira now ships structured workflow guidance as MCP Prompts in addition to its existing tools and resources. Three starter skills are included: feedback (sanitized bug report composer), wp-aware-editing (canonical edit workflow), and build-a-block-theme (end-to-end block theme creation). Site owners and plugins can register their own skills by adding a SKILL.md file under includes/skills/<id>/.

v1.3.0 - 2026/05/16
* New: Open Mira now provides an IDE-class WordPress development surface for AI agents, covering theme/page work, plugin fixes, hook navigation, sandbox plugin creation, and visual screenshot-driven iteration.
* New: WP-aware patch grammar for `theme.json` lets agents update design-system values with WordPress-specific selectors and bulk path merges instead of fragile full-file rewrites.
* Changed: `openmira/screenshot-url` creates minimal external Playwright capture jobs. Captured images are stored under `wp-content/openmira-screenshots/` for human or CI inspection; agent-visible screenshots should use the MCP client’s native browser/vision tooling.
* New: Project map, search, hook navigation, template resolution, linting, WP-CLI, URL probing, theme scaffolding, block scaffolding, and sandbox plugin promotion abilities are available for WordPress-aware development workflows.
* New: Audit Log now stores compacted full diffs for file-changing abilities and shows expandable diffs alongside backup IDs and restore points.
* New: Per-ability capability filters let site owners customize who may run each Open Mira ability.
* New: Production hard-block support via `OPENMIRA_BLOCK_PRODUCTION` and filterable environment detection.
* New: `execute-php` now has per-user rate limits and memory-delta protection to reduce runaway-agent risk.
* Improvement: Search-code now rejects or caps overly broad scans unless explicitly allowed, preventing runaway project searches.
* Improvement: CI now runs the wp-env smoke suite on push and pull request, covering the productization safety surface.
* Minor fixes

v1.2.0 - 2026/05/13
* New: AI agents can now upload plugin, theme, media, and other large files more easily.
* New: A "paste to your AI agent" block on the Configuration page lets you copy a short description and have your AI client set up the MCP server connection for you, instead of editing JSON config files by hand.
* New: The Configuration page warns when AI Abilities are turned on against what looks like a production site, with a reminder to keep them on staging or development copies only.
* New: If you already have a saved application password (for example from a password manager), a new field on the Configuration page lets you paste it directly and skip generating a new one.
* New: Local HTTPS development sites with self-signed certificates are detected automatically, and the generated client configs include the small flag Node.js needs to accept the certificate.
* New: When the site is on a local hostname over HTTP, the Application Passwords notice now explains the exact line to add to wp-config.php to enable them.
* Tweak: Redesigned the Configuration page into three numbered cards (Enable AI Abilities, Application Password, Connect Your AI Client) so the setup reads top to bottom.
* Tweak: The MCP endpoint is now /wp-json/mcp/openmira. The previous /wp-json/mcp/mcp-adapter-default-server keeps working for AI clients that are already connected, no action needed.
* Tweak: Per-client JSON snippets are collapsed by default behind a "Need the JSON config for a specific client?" link, since most setups now use the paste-to-agent block.
* Tweak: AI agents working with data-modeling plugins (ACF, JetEngine, Pods, ACPT, Meta Box, Toolset, CPT UI, WooCommerce, etc.) are now told to use the plugin's own UI or API instead of registering post types, taxonomies, or fields directly in PHP, avoiding broken slugs and labels later.
* Fix: PHP notices about duplicate ability registration on every wp-cron tick no longer appear in the error log.
* Minor Fixes
