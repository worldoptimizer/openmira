# Open Mira 1.7 Block-Level Editing Design

Status: design approved; spike passed; dynamic-path implementation in progress. This document records the architecture for Open Mira 1.7.0 and the implementation decisions made during the release.

## Reference Study

I studied the public GravityKit Block MCP reference under `#reference/block-mcp/` and the allowed upstream NovaMira Gutenberg files on `upstream/main`. I did not consult any Pro reference.

### Concepts to adopt from Block MCP

- Stable block references are the core primitive. Path and index addressing are useful for display, but they become stale when siblings are inserted, deleted, or moved.
- A block read should return a tree that includes each block's name, attributes, path, stable ref, and nested children so an agent can plan the next edit from one response.
- Batch writes must validate all operations before saving. A stale ref, duplicate target, invalid path, bad insert payload, or unsafe static-block operation aborts the whole batch.
- A successful batch should save exactly once so WordPress creates one revision for the user-visible change.
- ETag/If-Match style optimistic concurrency is the right shape. Open Mira will use an ability input field rather than depending on MCP clients setting HTTP headers.
- Move is deceptively complex because destination indexes are usually expressed against the pre-move tree. It must be atomic and must reject moves into the source or descendants.
- Gutenberg `innerContent` null placeholders are structural data. Insert, delete, and move operations must preserve or rebuild them consistently.

### Concepts not to adopt from Block MCP

- No separate Node/TypeScript MCP server. Open Mira remains a pure WordPress plugin using the existing Abilities/MCP adapter.
- No silent ref persistence on read. Block MCP can assign refs during reads; Open Mira's hard constraint is that reads are side-effect-free.
- No Yoast-specific behavior, preference system, translations, or broad endpoint surface in this release.

### Concepts to adopt from upstream NovaMira

- There are two Gutenberg write paths:
  - Dynamic/server-rendered blocks can be serialized safely in PHP when no static saved markup is required.
  - Static/native blocks need a browser running Gutenberg's JavaScript serializer; PHP should not reimplement each block's `save()` function.
- A browser runtime can be represented as an admin page with heartbeat/status, claim, complete, and fail endpoints.
- Runtime status should be visible to agents, with clear instructions when the page is offline.
- Pending static work should record the base content hash and fail cleanly on conflict.

### Concepts not to adopt from upstream NovaMira

- No SSE finalizer. Open Mira should use polling and a persistent admin tab, matching the already-validated runner-tab UX pattern.
- No coarse replace-content-only static MVP. When the static runtime ships, it should queue per-block operations instead of replacing the whole post.
- No namespace or product-shape changes.

## Existing Open Mira Foundation

Open Mira already has useful pieces:

- `openmira/read-gutenberg-content` returns post summaries and optional raw/parsed block data.
- `openmira/write-gutenberg-content` performs guarded full `post_content` replacement and must remain unchanged.
- `openmira/patch-gutenberg-blocks` already provides path-based append/prepend/replace/remove/update-attrs, but it is not stable-ref addressed and cannot safely handle static-block serializer concerns.
- `openmira/create-gutenberg-serialization-job` and the Block Tools admin page already load Gutenberg editor scripts and call `wp.blocks.serialize()`. The static runtime should reuse this direction rather than starting from zero.
- `openmira/apply-patch` already has a parser dispatch that can accept new hunk types.
- Plan/Act mode, audit logging, builder backups, stale-write hashes, smoke tests, and Playwright CI are established patterns.

## Proposed Ref Persistence Mechanism

Decision: use in-content `_openmira_ref` attributes, but persist them only on writes.

Reads stay side-effect-free:

- `openmira/read-blocks` parses the current `post_content`.
- If a block already has `attrs.metadata._openmira_ref`, the response returns that as a durable `ref` with `ref_source: "persisted"`.
- If a block lacks a ref, the response returns an ephemeral virtual ref with `ref_source: "virtual"`.
- A virtual ref is valid only against the returned `etag`. It lets an agent perform the first patch after an initial read without mutating content during the read.

Writes normalize refs:

- `openmira/patch-blocks` resolves each target against the current parse tree and the caller's `expected_etag` when supplied.
- Before a successful save, Open Mira assigns a durable UUID to every block in the saved tree that lacks one.
- The ref is stored in `attrs.metadata._openmira_ref`.
- This produces one normal WordPress revision as part of the user's requested write, not a hidden metadata write.

Rationale:

- Sidecar post meta keeps markup pristine, but it cannot reliably survive manual Gutenberg edits, block copy/paste, or nested tree rewrites because the identity is not carried with the block itself.
- In-content metadata is Gutenberg-native and is expected to round-trip through parse and serialize. The Step 1 spike must prove this for `_openmira_ref`; if it fails, implementation stops.
- Persist-on-write gives agents durable refs after the first mutation while preserving the invariant that reads never modify stored content.

### Virtual refs

Virtual refs should be namespaced so they cannot be mistaken for durable refs:

```text
v:<path>:<short-content-hash>
```

The path component makes the virtual ref resolvable in the current tree. The short hash reduces accidental matches after content shifts. A virtual ref can be accepted only when `expected_etag` matches the read response that produced it.

### Duplicate refs from copy/paste

Gutenberg copy/paste is likely to duplicate block attributes, including `_openmira_ref`. Policy:

- `read-blocks` must detect duplicate persisted refs.
- The first occurrence can keep the durable ref.
- Later occurrences are returned with a generated virtual ref and a warning such as `duplicate_persisted_ref`.
- A write targeting an ambiguous durable ref must fail until the agent re-reads and targets the disambiguated virtual ref with the matching `expected_etag`.
- On any successful write, duplicate persisted refs are rewritten to fresh durable refs before saving.

This policy avoids a read-side mutation while preventing a copied block from sharing identity with its source forever.

## Maintainer Review Decisions

- Virtual refs are approved for `patch-blocks`, but only when paired with `expected_etag`.
- `move` is not a release blocker. Build and stabilize `update`, `insert`, and `delete` first; attempt `move` after those are solid. If same-parent, cross-parent, and `innerContent` placeholder tests are not stable, defer `move` to a fast-follow release.
- `patch-blocks` is the primary agent-facing surface. Static-block work should route internally to a runtime job and expose a poller such as `get-block-patch-job` only when needed.
- Keep `openmira/patch-gutenberg-blocks` registered for backwards compatibility, but demote it from compact discovery after `openmira/patch-blocks` ships.

## Step 1 Spike

The spike must run before production implementation.

Required deterministic smoke:

1. Create a post containing nested Gutenberg blocks.
2. Assign `_openmira_ref` in `attrs.metadata` in memory.
3. Serialize with `serialize_blocks()`, save via `wp_update_post()`, re-read, parse, and assert refs survive at the same logical blocks.
4. Copy/duplicate a block in the parsed tree, serialize, re-parse, and assert duplicate detection policy can identify the duplicate.

Manual gating assertion:

- Open the post in the Gutenberg editor, make an unrelated text edit, save, and re-run `read-blocks`.
- Copy/paste a block in the editor and record whether the ref is duplicated, stripped, or transformed.

Hard stop:

- If `_openmira_ref` does not survive a normal Gutenberg editor save, stop and report the symptom. Do not build `read-blocks` or `patch-blocks` on unstable refs.
- The editor-save assertion is as important as the PHP smoke. PHP `serialize_blocks()` preserving the key is necessary but not sufficient; the real unknown is whether the Gutenberg editor preserves arbitrary custom keys under `metadata` during an open/edit/save cycle.

## Abilities

### `openmira/read-blocks`

Read-only, side-effect-free.

Input:

- `post_id`
- `max_depth` default 4, capped at 12
- `include_attributes` default true
- `include_inner_html` default false
- `include_raw_content` default false

Output:

```json
{
  "post": {"id": 123, "title": "Home", "post_type": "page", "modified": "..."},
  "etag": "m:2026-05-28T12:34:56Z",
  "content_hash": "sha256...",
  "has_duplicate_refs": false,
  "blocks": [
    {
      "ref": "omr_...",
      "ref_source": "persisted",
      "path": [0],
      "name": "core/group",
      "dynamic": false,
      "attributes": {},
      "inner_html_length": 128,
      "inner_block_count": 2,
      "innerBlocks": []
    }
  ],
  "warnings": []
}
```

`etag` is derived from `post_modified_gmt`. It is intentionally human-readable in responses but treated as opaque by callers.

### `openmira/patch-blocks`

Destructive, Plan/Act gated, audited, backed up.

Input:

- `post_id`
- `operations[]`
- `expected_etag` optional but strongly recommended
- `create_backup` default true
- `include_blocks` default true for compact post-save verification

Operation shape:

```json
{"op": "update", "ref": "omr_...", "attrs": {"metadata": {"name": "Hero"}}}
{"op": "insert", "after": "omr_...", "block": {"name": "openmira/example", "attrs": {}, "innerBlocks": []}}
{"op": "delete", "ref": "omr_..."}
{"op": "move", "ref": "omr_...", "after": "omr_..."}
```

Validation:

- Resolve all refs first.
- Reject stale or ambiguous refs.
- Reject duplicate targets unless the operation sequence explicitly depends on a prior operation's inserted ref; v1 should avoid that complexity and require each op to target the pre-batch tree.
- Reject invalid insertion payloads.
- Reject operations that would exceed a block-depth cap.
- Reject static-block mutations that require browser serialization when using the PHP fast path.
- Validate `expected_etag` before any backup or save.

Save semantics:

- Apply operations in memory.
- Normalize missing and duplicate refs.
- Serialize and save once.
- Return one pre-write backup, one previous/new hash pair, one previous/new ETag pair, and one WordPress revision.

### Dynamic-block fast path

The PHP fast path should be conservative:

- Safe when every touched, inserted, or moved block is dynamic/server-rendered and does not rely on static saved DOM.
- Dynamic detection uses `WP_Block_Type_Registry::get_instance()->get_registered($name)` and the registered block type's dynamic/server-rendered signal.
- Open Mira-owned scaffolded dynamic blocks should pass.
- Core/static blocks should fail with `block_runtime_required` and a response that explains how to use the Block Editor Runtime.

This may initially cover fewer blocks than theoretically possible. Conservatism is preferable to invalid Gutenberg markup.

### Static-block runtime path

Static operations use a new Block Editor Runtime admin page.

Planned shape:

- `openmira/patch-blocks` detects static-block-required operations and creates a pending block-patch job instead of direct PHP save.
- The response includes `job_id`, `runtime_url`, `runtime_status`, and instructions to keep the runtime tab open.
- The runtime page polls for jobs assigned to the current user.
- For each job, it loads the target post in an editor-capable context, parses the current content with `wp.blocks.parse()`, applies the validated operations in JavaScript, serializes with `wp.blocks.serialize()`, and posts completion back to WordPress.
- The completion endpoint performs a final ETag check, creates the backup, writes the serialized content once, and records success or conflict.
- Failed jobs retain a structured error visible through a read-job ability.

This reuses the proven persistent-runner-tab model: open once, poll every 1-2 seconds, process sequentially, show visible queue state, no SSE.

Scope risk:

- If editor-context serialization proves too fragile for 1.7.0, ship the dynamic fast path plus documented static runtime design and defer Step 4. That decision must be explicit and evidence-backed.

## Patch Grammar Integration

Extend `openmira/apply-patch` by adding new parser-dispatched hunk types.

Example:

```text
*** Begin Patch
*** Update Block (post_id: 123, ref: omr_abc, expected_etag: m:2026-05-28T12:34:56Z):
{
  "attrs": {
    "metadata": {"name": "Hero"}
  }
}
*** Insert Block (post_id: 123, after: omr_abc):
{
  "name": "openmira/callout",
  "attrs": {"title": "Next step"},
  "innerBlocks": []
}
*** Delete Block (post_id: 123, ref: omr_old):
*** End Patch
```

Parser plan:

- Add block hunk header recognition alongside the current `theme-json` dispatcher.
- Convert block hunks into a grouped `patch-blocks` call per `post_id`.
- Require one `expected_etag` per grouped post patch if any hunk supplies it; reject conflicting ETags in one envelope.
- Reuse existing `dry_run`, `include_diff`, and audit semantics where they fit. Block patches should return operation summaries rather than file diffs.

## Async WP-CLI

This subsystem is independent and can proceed in parallel after the design is accepted.

Changes:

- `openmira/run-wpcli` gains `mode: "sync" | "async"` with default `sync`.
- Existing input remains valid.
- Async mode writes job state under `wp-content/openmira-wpcli-jobs/<job_id>/`.
- Add `openmira/get-wpcli-job` with `job_id` and optional `log_offset`.
- `run-wpcli` annotation should be corrected to `destructive: true`; even read-like WP-CLI commands execute arbitrary plugin/bootstrap code.

Job files:

- `job.json` for args, command, status, timestamps, pid, exit code, policy, and user ID.
- `stdout.log` and `stderr.log` or one combined `output.log`.
- `done.json` written atomically when the process exits.

Process model:

- Reuse the existing allowlist and Plan/Act policy before spawning.
- Build commands with `escapeshellarg`.
- Refuse root execution unless `OPENMIRA_WPCLI_ALLOW_ROOT` is defined and truthy.
- Use a bounded runtime and record timeout status.
- Opportunistically delete jobs older than 24 hours.

Polling response:

```json
{
  "job_id": "wpcli_...",
  "status": "running",
  "exit_code": null,
  "log": "new bytes since offset",
  "log_offset": 1234,
  "next_log_offset": 4567
}
```

## Tests

- `tests/smoke/block-ref-roundtrip.php`: proves PHP parse/serialize/save/reparse ref survival and duplicate-ref detection.
- `read-blocks`: tree shape, ETag, virtual refs on untagged content, side-effect-free read by comparing content hash before/after.
- `patch-blocks`: dynamic update/insert/delete success, stale ref abort, duplicate target abort, ETag conflict, one revision per batch, and static-runtime-required refusal.
- `apply-patch`: block hunk parsing and dispatch.
- `run-wpcli`: sync back-compat, async lifecycle, log offset, root refusal when applicable.

Playwright smoke for the deferred static runtime:

- Block Editor Runtime page renders in wp-env.
- Queue a static-block patch, leave the runtime tab open, assert completion.
- Reopen the post editor and assert no Gutenberg invalid-content warning is visible.

Manual evidence:

## Built-In Skill

Add `includes/skills/block-editing/SKILL.md` with `enable_prompt: true`.

The skill should teach:

- Start with `openmira/read-blocks`.
- Prefer `openmira/patch-blocks` for localized edits.

## Reviewed Decisions

- Virtual refs are accepted in `patch-blocks` only with `expected_etag`; this preserves side-effect-free reads without forcing a separate seeding write.
- `move` is deferred from the first implementation. Agents can use delete plus insert until direct move coverage is stable.
- `patch-blocks` is the primary agent-facing surface. Static-block job helpers should only be added when the browser runtime exists.
- `openmira/patch-gutenberg-blocks` stays registered for backwards compatibility but is demoted from compact discovery.

## Implementation Checkpoint — 2026-05-28

The stable-ref spike passed. A real Gutenberg editor open → edit → save cycle preserved `attrs.metadata._openmira_ref` on all pre-existing seeded blocks, so the in-content persisted-on-write mechanism is viable and the sidecar-meta fallback is not needed.

The first implementation slice ships the dynamic path:

- `openmira/read-blocks`
- `openmira/patch-blocks` for dynamic-block update, insert, and delete
- block hunks in `openmira/apply-patch`
- async `openmira/run-wpcli`
- `block-editing` built-in Skill

The browser-backed static Block Editor Runtime is explicitly deferred. Static/core block operations return `block_runtime_required` rather than producing PHP-guessed saved markup. The decision record lives at `docs/pilots/1.7.0-block-editing/static-runtime-decision.md`.
